What is Claude Code?

Think: "Ahrefs but for your codebase"

Claude Code is an AI-powered coding assistant that lives in your terminal (or IDE). Just like Ahrefs crawls websites to find SEO issues and opportunities, Claude Code reads your entire codebase to understand it — then helps you write, fix, and improve code through natural conversation.

But here's the key difference from ChatGPT: Claude Code has direct access to your files. It can read them, edit them, run terminal commands, and even make git commits. It's not just a chat — it's an agent that takes action.

What makes it different?

  • Agentic — it doesn't just suggest code, it writes and edits files directly
  • Context-aware — it reads your project files like Googlebot crawls your site
  • Tool-using — it runs terminal commands, searches files, and manages git
  • Persistent — CLAUDE.md and memory files carry context between sessions (like your site's link equity persists across crawls)
Try It

You're already using Claude Code right now! Ask it: "What files are in this project?" — watch it explore your codebase like a crawler discovering pages.

Pro Tip

Claude Code is available as a CLI, desktop app, web app (claude.ai/code), and IDE extensions for VS Code and JetBrains. Pick whatever feels most natural to start with — you can always switch later.

Section 3

Core Concepts: Prompts, Context & CLAUDE.md

Think: "CLAUDE.md = robots.txt; Context window = crawl budget"

In SEO, three things determine how well Google understands your site: robots.txt (what to crawl), crawl budget (how much to crawl), and site architecture (how content is organized). Claude Code has direct equivalents.

CLAUDE.md = Your robots.txt + meta directives

Just like robots.txt tells Googlebot how to behave on your site, CLAUDE.md tells Claude Code how to behave in your project. It's a markdown file at your project root that contains:

  • Project description and purpose
  • Coding conventions and rules
  • File structure explanations
  • Behavioral directives (how Claude should respond)
# CLAUDE.md — Your project's "robots.txt" for AI

## Project Identity
This is a learning lab for AI implementation.

## Rules
- Use SEO analogies when explaining concepts
- Always explain the "why" behind changes
- Keep code simple and well-commented

Context Window = Crawl Budget

Google allocates a "crawl budget" to your site — a limit on how many pages it will crawl per session. Claude Code has a context window — a limit on how much information it can hold at once. You want to use it efficiently:

  • Don't dump everything — just like you wouldn't want Googlebot wasting crawl budget on thin pages
  • Be specific — point Claude to the right files, like using a sitemap to guide crawlers
  • CLAUDE.md is always loaded — it's like your homepage, always crawled first

Prompts = Search Queries

The way you talk to Claude Code is exactly like crafting search queries. Vague queries get vague results:

# Bad (like searching "shoes") — too vague
"fix this code"

# Good (like searching "best running shoes for flat feet under $100")
"Fix the mobile menu toggle in experiments/01-portfolio/script.js —
 the nav-links don't close when I click a link on mobile"
Try It

Ask Claude Code: "Read the CLAUDE.md file and tell me what you know about this project." Watch how it uses CLAUDE.md as its first source of truth — just like Googlebot reads robots.txt first.

Pro Tip

Your CLAUDE.md is the highest-leverage file in your entire project. A well-written CLAUDE.md is like having perfect technical SEO — everything else works better because the foundation is right.

Section 5

The Permission Model

Think: "noindex/nofollow — controlling what the AI can touch"

In SEO, you use noindex and nofollow to control what Google can access and follow. Claude Code has a similar permission system — you control what it's allowed to do.

How Permissions Work

  • Read files — always allowed (like Googlebot reading public pages)
  • Edit files — asks for approval each time (like staging changes before publish)
  • Run commands — asks for approval (some commands are safe, others are destructive)
  • Git operations — asks before committing, pushing, etc.

Pre-approving Commands

You can pre-approve safe commands in .claude/settings.json — like adding pages to your sitemap so they're always crawled:

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(python -m http.server*)",
      "Bash(git status*)",
      "Bash(git log*)",
      "Bash(git diff*)"
    ],
    "deny": []
  }
}

Permission Modes

When Claude asks for permission, you can:

  • Allow once — approve this specific action
  • Allow always — auto-approve this type of action going forward
  • Deny — block the action (like adding a disallow rule)
Try It

Ask Claude Code to edit a file — notice how it asks for your permission before making changes. Then try: "Show me the current .claude/settings.json" to see your permission configuration.

Pro Tip

Start restrictive (approve everything manually) and gradually open up permissions as you gain confidence — just like you'd start with a restrictive robots.txt and expand access as you verify things work.

Advanced Prompting

Think: "Long-tail keywords = specific prompts get better results"

In SEO, "running shoes" gets you millions of competing results. "Best trail running shoes for flat feet under $150" gets you exactly what you need. Prompting Claude Code works the same way.

The Specificity Ladder

# Level 1: Generic (like "shoes") — weak results
"Make this better"

# Level 2: Directional (like "running shoes") — okay results
"Improve the CSS performance"

# Level 3: Specific (like "trail running shoes size 10") — good results
"Reduce the CSS file size in experiments/01-portfolio/style.css
 by combining duplicate rules and removing unused selectors"

# Level 4: Expert (like a full keyword brief) — best results
"Audit experiments/01-portfolio/style.css for performance:
 1. Find and merge duplicate CSS rules
 2. Remove any selectors not used in index.html
 3. Convert px values to rem where appropriate
 4. Add will-change hints for animated properties
 Report what you changed and why."

Prompt Patterns That Work

  • Role + Task + Constraints — "As a senior developer, review this code for security issues. Focus only on user input handling."
  • Step-by-step — "First read the file, then identify issues, then fix them one by one."
  • Show don't tell — Paste an example of what you want, then say "do this for all similar files."
  • Negative constraints — "Don't add new dependencies. Don't change the visual appearance."
Try It

Take a task you'd normally do manually and write it as a Level 4 prompt. For example: "Audit all HTML files in experiments/ for proper heading hierarchy (only one H1, logical H2-H6 nesting). List violations in a table, then fix them."

Pro Tip

The best prompts include both what you want AND what you don't want. Just like a good SEO brief specifies target keywords AND keywords to avoid cannibalizing.

Plan Mode

Think: "SEO audit before making changes — analyze first, act second"

You'd never start implementing SEO changes without first doing an audit and creating a strategy. Plan Mode is exactly that — it forces Claude Code to analyze and plan before touching any files.

How Plan Mode Works

# Activate plan mode (in Claude Code CLI)
/plan

# Or type "plan" as part of your prompt
"Plan how you would add a contact form to the portfolio site.
 Don't make any changes yet — just give me the strategy."

# Claude will:
# 1. Explore relevant files (the audit)
# 2. Write a plan file (the strategy doc)
# 3. Present it for your approval
# 4. Only execute after you approve

When to Use Plan Mode

  • Complex changes — anything touching 3+ files (like a site-wide SEO migration)
  • Learning new concepts — ask Claude to plan, then study the plan to learn
  • Risky operations — when you want to review before any changes happen
  • Team alignment — share the plan with others before executing
Try It

Ask Claude Code: "Plan how you would restructure the dental clinic site to improve its SEO architecture — better URL structure, internal linking, and content hierarchy. Don't change anything yet." Review the plan it creates.

Pro Tip

Plan mode is also a great learning tool. Ask Claude to plan something complex, then study the plan to understand the approach. It's like reading an SEO audit report to learn what to look for.

Working with Git

Think: "Commits = indexing; Branches = staging environments"

When Google indexes your page, that version is "live" and discoverable. Git commits work the same way — they create a permanent, discoverable snapshot of your code. And just like you might test SEO changes on a staging site before pushing to production, git branches let you experiment safely.

Essential Git Operations via Claude Code

# Check status (like checking your indexation report)
"Show me git status — what's changed since last commit?"

# Commit changes (like submitting pages for indexing)
"Commit these changes with message: Add meta descriptions to all pages"

# Or use the slash command
/commit

# View history (like checking your crawl history)
"Show me the last 5 commits"

# Create a branch (like setting up a staging environment)
"Create a new branch called 'add-contact-form' and switch to it"

# Compare branches (like comparing staging vs production)
"Show me what's different between this branch and main"

The Git Workflow (SEO Analogy)

  • Working directory = Your draft content (not published yet)
  • Staging (git add) = Content review queue (approved but not live)
  • Commit = Publishing / indexing (permanent and discoverable)
  • Branch = Staging site (experiment without affecting production)
  • Merge = Deploying staging to production
Try It

Ask Claude Code: "Show me the git log for this project." You should see the commits from when we restructured Project Alpha. Then try: "Create a new branch called 'experiment', make a small change, commit it, and switch back to main."

Pro Tip

Use the /commit slash command for quick commits. Claude Code will analyze your changes, draft a good commit message, and handle staging — like having a content editor who writes your meta descriptions for you.

Section 10

MCP Servers

Think: "API integrations — like connecting Ahrefs to Search Console"

In SEO, tools become powerful when they connect: Ahrefs pulling Search Console data, Screaming Frog connecting to Google Analytics, SEMrush integrating with your CMS. MCP (Model Context Protocol) does the same for Claude Code — it lets you plug in external tools and data sources.

What MCP Does

MCP is a protocol that lets Claude Code talk to external services through "servers" — think of each MCP server as a plugin or API integration:

  • File system server — gives Claude deeper file access
  • GitHub server — Claude can create PRs, manage issues, review code
  • Database servers — Claude can query your databases directly
  • Custom servers — you can build your own (we'll cover this in the MCP learning module)

How to Connect an MCP Server

# MCP servers are configured in your Claude Code settings
# Example: adding a filesystem MCP server

// In .claude/settings.json or via the /mcp command
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/dir"]
    }
  }
}

The Big Picture

MCP is what turns Claude Code from a coding assistant into an AI implementation platform. Just like SEO tools evolved from standalone crawlers to integrated suites, Claude Code + MCP becomes your integrated AI development environment.

Try It

Ask Claude Code: "What MCP servers are currently connected? List any available tools." Then explore: "Explain what MCP servers would be most useful for a web developer."

Pro Tip

Don't rush to add MCP servers. Start with Claude Code's built-in tools — they cover 90% of use cases. Add MCP servers when you hit a specific need, just like you'd add a new SEO tool only when your current stack has a gap.

Multi-File Projects

Think: "Managing SEO across a multi-page site"

Optimizing a single page is easy. Managing SEO across a 500-page site? That requires strategy. Same with Claude Code — working with multiple files requires understanding how to guide the AI across your project.

Techniques for Multi-File Work

# Point Claude to specific files (like prioritizing crawl)
"Look at experiments/02-dental-clinic/style.css and
 experiments/02-dental-clinic/script.js together.
 Find any style conflicts."

# Batch operations (like bulk meta description updates)
"Add a consistent footer to ALL HTML files in
 experiments/02-dental-clinic/"

# Cross-file refactoring (like site-wide internal linking)
"The dental clinic pages each define their own nav HTML.
 Extract the nav into a reusable pattern and make all
 pages consistent."

# Using glob patterns (like URL patterns in robots.txt)
"Find all .html files that are missing a viewport meta tag"

CLAUDE.md for Multi-File Context

For large projects, your CLAUDE.md becomes critical — it's the sitemap that helps Claude efficiently navigate:

  • Document your folder structure
  • Explain relationships between files
  • Note shared resources and dependencies
  • Flag files that shouldn't be modified
Try It

Ask Claude Code: "Compare the CSS architecture across all three experiments. Which patterns are shared? Which are unique? Create a summary table." This exercises Claude's ability to reason across multiple files.

Pro Tip

When working across many files, break the task into phases: first analyze, then plan, then execute in groups. Just like you'd segment a large site migration into phases.

Slash Commands & Workflows

Think: "Saved audit configs — one-click recurring workflows"

In Screaming Frog, you save crawl configurations so you don't have to set up the same filters every time. Claude Code has slash commands — quick shortcuts for common workflows.

Built-in Slash Commands

/help          — Get help with Claude Code features
/commit        — Smart commit: analyzes changes, drafts message, commits
/plan          — Enter plan mode (analyze before acting)
/review-pr     — Review a pull request
/clear         — Clear the conversation (free up context window)
/fast          — Toggle fast output mode (same model, faster)

Custom Workflows with Skills

Beyond built-in commands, Claude Code supports skills — think of them as custom audit templates:

# Skills extend Claude Code's capabilities
# They're like Screaming Frog custom extraction rules

# Some skills you can use:
/simplify      — Review changed code for quality and efficiency
/loop          — Run a task on a recurring interval

Hooks — Automated Workflows

Hooks let you run shell commands automatically in response to Claude Code events — like setting up automated Lighthouse audits that run after every deployment:

// In settings.json — run a linter after every file edit
{
  "hooks": {
    "afterEdit": {
      "command": "echo 'File edited: $FILE_PATH'"
    }
  }
}
Try It

Try these commands: /help to see available commands, then make a small change to any file and use /commit to see the smart commit flow in action.

Pro Tip

Use /clear liberally. When you switch tasks, clear the context — it's like starting a fresh crawl session instead of carrying stale data from the last audit.

Tips & Tricks for Daily Use

Think: "Power-user SEO habits that save hours"

Every SEO pro has workflows that save them hours — keyboard shortcuts in Screaming Frog, saved segments in Analytics, template reports. Here are the Claude Code equivalents.

Efficiency Habits

  • Start every session by reading CLAUDE.md — ask Claude "What do you know about this project?" to verify its understanding
  • Use /clear between tasks — don't let stale context pollute new work
  • Be specific about scope — "Fix the nav in index.html" beats "fix the nav"
  • Review before approving — always read the diff before accepting edits
  • Commit often — small commits are like incremental indexing, easy to track and roll back

Learning Habits

  • Ask "why" — after Claude makes a change, ask "Why did you choose this approach?" to deepen understanding
  • Ask for alternatives — "What's another way to solve this?" teaches you patterns
  • Use Plan Mode for learning — plan complex tasks, study the plan, then decide if you want to execute
  • Keep a learning log — note down new concepts in the learning/ folder

Debugging Habits

  • When Claude gets stuck — rephrase your request, add more context, or point to the specific file
  • When edits break things — use git to roll back: "Undo the last commit"
  • When context feels stale — /clear and start fresh with a focused prompt
Try It

Practice the learning habit: Ask Claude to make any change, then ask "Why did you do it that way? What alternatives did you consider?" — this is how you learn the fastest.

Pro Tip

The memory system (CLAUDE.md + .claude/memory/) is your most powerful tool. Invest time in writing good CLAUDE.md files — the ROI compounds with every session, just like domain authority compounds over time.
