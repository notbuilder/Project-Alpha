Chapter 1

What is Git?

Think: "The Wayback Machine for your code"

Git is a version control system. It tracks every change you make to your files and lets you travel back in time to any previous version. It runs entirely on your computer — no internet needed.

Without Git, your project history looks like this:

project-final.html
project-final-v2.html
project-FINAL-FINAL.html
project-backup-march.html
project-dont-touch.html

With Git, you have one file and a complete history of every change:

project.html
commit a1b2c3 — "Add contact form" (March 15)
commit d4e5f6 — "Fix mobile nav" (March 14)
commit g7h8i9 — "Initial layout" (March 12)
SEO Analogy

Git is like Google's cache of your website. Google keeps snapshots of your pages over time. If something breaks, you can check the cached version. Git does the same for code — every "commit" is a cached snapshot you can restore anytime.

Why Git Matters

  • Undo anything — broke something? Roll back to the last working version in seconds
  • Work in parallel — try risky experiments on a "branch" without affecting your main code
  • Collaborate — multiple people can work on the same project without overwriting each other
  • Audit trail — see who changed what, when, and why
Chapter 2

What is GitHub?

Think: "Google Drive for your code, but with superpowers"

Git = the tool on your computer that tracks changes.
GitHub = a website that hosts your Git repositories online so you can share, collaborate, and back up your code.

Your Computer (Local) GitHub (Remote)
┌───────────────────┐ ┌──────────────────┐
│ Your files │ → push → │ Cloud backup │
│ Git history │ ← pull ← │ Collaboration │
│ Branches │ │ Pull requests │
└───────────────────┘ └──────────────────┘
SEO Analogy

Git is like Screaming Frog (runs locally on your machine). GitHub is like Ahrefs (cloud-based, accessible anywhere, enables collaboration). You can use Git without GitHub, just like you can crawl a site locally without Ahrefs — but the cloud version adds sharing, backups, and team features.

Other Platforms Like GitHub

GitHub is the most popular, but alternatives exist: GitLab, Bitbucket, Azure DevOps. They all work with Git — think of them as different CDNs serving the same content.

Working Directory → Staging Area → Local Repository → Remote (GitHub)

"Your draft" "Review queue" "Published" "Live on CDN"

You edit files → git add → git commit → git push
here stages them saves snapshot uploads to GitHub

Chapter 4

Essential Git Commands

Think: "Your daily SEO toolkit — the 10 commands you'll use 90% of the time"

Setup (One-Time)

Terminal 
# Tell Git who you are (like setting up your Google profile)
git config --global user.name "Shoaib Tariq"
git config --global user.email "your@email.com"

# Create a new Git repository in the current folder
git init

# Or download an existing repo from GitHub
git clone https://github.com/username/repo-name.git

The Daily Workflow

Terminal 
# 1. Check what's changed (like running a site crawl)
git status

# 2. See the actual changes line-by-line (like diffing page versions)
git diff

# 3. Stage specific files for commit (add to review queue)
git add index.html style.css

# 4. Or stage everything that changed
git add -A

# 5. Commit with a descriptive message (publish the snapshot)
git commit -m "Fix mobile nav and add meta descriptions"

# 6. Push to GitHub (deploy to the CDN)
git push

Viewing History

Terminal 
# See commit history (like checking your indexation log)
git log --oneline

# Output:
# a1b2c3d Fix mobile nav and add meta descriptions
# e4f5g6h Add contact form to homepage
# i7j8k9l Initial commit

# See who changed a specific line in a file (blame = audit trail)
git blame index.html

# See what changed in a specific commit
git show a1b2c3d

Undoing Things

Terminal 
# Discard changes to a file (revert to last commit)
git checkout -- index.html

# Unstage a file (remove from review queue, keep changes)
git restore --staged index.html

# Undo the last commit but keep the changes
git reset --soft HEAD~1

# Create a NEW commit that reverses a previous one (safe undo)
git revert a1b2c3d
Caution

git reset --hard permanently deletes uncommitted changes. It's like deleting pages from your CMS with no recycle bin. Always prefer git revert — it creates a new commit that undoes the old one, keeping your history intact.

Chapter 5

Branching: Parallel Universes for Your Code

Think: "Staging environments — experiment without breaking production"

A branch is a copy of your code that you can modify independently. When you're happy with the changes, you merge it back. If it was a mistake, you delete the branch — main is untouched.

main •——•——•——•—————•——•
\ /
feature •—•—•
(work here) (merge back)
Terminal 
# Create a new branch and switch to it
git checkout -b add-contact-form

# You're now on "add-contact-form" branch
# Make changes, commit as normal...
git add .
git commit -m "Add contact form with validation"

# Switch back to main
git checkout main

# Merge the feature branch into main
git merge add-contact-form

# Delete the branch (it's merged, no longer needed)
git branch -d add-contact-form
SEO Analogy

Branches are like staging environments. You'd never test a site redesign on the live production URL — you'd use a staging subdomain, verify everything works, then deploy. Git branches work exactly the same way: experiment in isolation, merge when ready.

experiment/experiment/dark-themeTrying something risky
Chapter 6

Working with GitHub (Remote)

Think: "Syncing your local SEO work to the cloud dashboard"

Connecting Local to GitHub

Terminal 
# Add a GitHub repo as your "remote" (like connecting Search Console)
git remote add origin https://github.com/notbuilder/Project-Alpha.git

# Push your code to GitHub for the first time (-u sets up tracking)
git push -u origin main

# After the first push, just use:
git push

Pulling Changes (Syncing)

Terminal 
# Download latest changes from GitHub (like refreshing your crawl data)
git pull

# See what's on GitHub without downloading yet
git fetch

Pull Requests (PRs)

A Pull Request is how you propose changes on GitHub. Instead of merging directly, you create a PR that others can review, comment on, and approve before merging.

1. Create branch locally → 2. Push branch to GitHub → 3. Open PR on GitHub
↓
4. Team reviews your code → 5. Approve & merge → 6. Branch deleted
Terminal (using GitHub CLI) 
# Push your branch to GitHub
git push -u origin add-contact-form

# Create a Pull Request from the command line
gh pr create --title "Add contact form" --body "Adds a validated contact form to the homepage"

# List open PRs
gh pr list

# Merge a PR
gh pr merge 42
SEO Analogy

A Pull Request is like submitting a guest post for editorial review. You write the content (code), submit it for review (PR), the editor checks it (code review), requests changes if needed, and finally publishes it (merge). Nobody's work goes live without review.

Chapter 7

Git Worktrees: Multiple Branches, Simultaneously

Think: "Having staging AND production open in different browser tabs"

Normally, you can only have one branch checked out at a time. Switching branches changes all your files. Git worktrees solve this — they let you check out multiple branches in separate folders simultaneously.

The Problem Worktrees Solve

Without worktrees:
1. Working on feature branch
2. Urgent bug reported on main
3. Must stash/commit half-done work
4. Switch to main, fix bug
5. Switch back, unstash
6. Context lost, flow broken

With worktrees:
1. Working on feature in /project
2. Urgent bug reported
3. Create worktree: /project-hotfix (on main)
4. Fix bug in the other folder
5. Never left your feature branch
6. Zero context switching

Worktree Commands

Terminal 
# Create a worktree — checks out "main" in a separate folder
git worktree add ../project-hotfix main

# Now you have TWO folders:
#   /project          ← your feature branch (untouched)
#   /project-hotfix   ← main branch (fix the bug here)

# List all worktrees
git worktree list

# Remove a worktree when done
git worktree remove ../project-hotfix

How Worktrees Actually Work

One .git database (shared)
│
┌─────────────────┐ ┌────────────────┐
│ /project │ │ /project-hotfix│
│ branch: feature │ │ branch: main │
│ Your WIP code │ │ Clean main code │
└─────────────────┘ └────────────────┘

Both folders share the SAME Git history and commits.
Changes committed in either folder are visible to both.
SEO Analogy

Worktrees are like having your production site and staging site open in separate browser tabs. Both are the same site, but showing different versions. You can work on staging without touching production — and when staging looks good, you merge it over. No tab-switching nightmares.

When to Use Worktrees

  • Hotfix while mid-feature — don't stash, don't lose context
  • Comparing two branches visually — open both folders in your editor
  • Running tests on main while still coding on your feature
  • Code reviews — check out a PR in a worktree to test it without switching your own branch

How Claude Code Leverages Git

Think: "Your AI assistant has built-in version control superpowers"

Claude Code doesn't just "know about" Git — it actively uses Git as part of its workflow. Understanding how helps you work with it more effectively.

Claude Code and Worktrees

This is where it gets powerful. When you give Claude Code a complex or risky task, it can use git worktrees for isolation:

Your main project Claude's worktree
┌──────────────────┐ ┌──────────────────┐
│ /Project Alpha │ │ /tmp/worktree-xx │
│ Your files stay │ │ Claude works │
│ exactly as they │ │ on a separate │
│ are. Untouched. │ │ copy. Safe. │
└──────────────────┘ └──────────────────┘

If the work succeeds → merge the worktree changes back
If it fails → delete the worktree, your project is unchanged

How Worktree Isolation Works in Claude Code

  1. Agent spawns with isolation: "worktree" — Claude creates a temporary git worktree in a separate directory
  2. All edits happen in the worktree — your actual project files are never touched during the experiment
  3. If the agent makes no changes — the worktree is automatically cleaned up, zero mess left behind
  4. If the agent makes changes — the worktree path and branch are returned, you choose whether to merge

This is why Claude Code can confidently attempt risky refactors, multi-file changes, and experimental approaches — if anything goes wrong, your real code is untouched.

SEO Analogy

This is exactly like how you'd test a major site migration. You'd never run a URL restructure on the live site first — you'd clone it to a staging environment, run the migration there, verify everything works, then deploy. Claude Code's worktree isolation is that staging environment. If the migration breaks, production is unaffected.

The /commit Slash Command

When you type /commit in Claude Code, here's what happens behind the scenes:

1. git status — what files changed?
2. git diff — what exactly is different?
3. git log --oneline — what's your commit message style?
4. AI analyzes the diff — drafts a descriptive commit message
5. git add + git commit — stages and commits
6. git status — verifies the commit succeeded

One slash command replaces a 6-step manual workflow. And because Claude reads your recent commit history first, the message style matches your existing pattern.

Git Safety in Claude Code

Claude Code follows strict Git safety rules:

  • Never force-pushes to main/master without explicit permission
  • Never amends existing commits unless asked — always creates new commits
  • Never skips hooks (no --no-verify) unless asked
  • Prefers specific git add over git add -A (avoids accidentally staging .env files or secrets)
  • Always asks before destructive operations like git reset --hard or branch deletion
Key Insight

Claude Code treats Git as a safety net, not just a tool. Every commit is a checkpoint you can return to. This is why Claude commits frequently — small, focused commits mean you can undo any single change without losing other work. Think of it like saving your game before a boss fight.
